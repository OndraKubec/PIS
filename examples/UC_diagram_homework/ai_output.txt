Specializované use casy napojíš přímo na konkrétní oddělení.
Volitelně můžeš přidat ještě:

2) Správné <<include>> vztahy (doporučené)
A) Plánování a spouštění výroby
Spustit výrobní plán <<include>> Naplánovat výrobní zakázky
Dává smysl, pokud „spuštění plánu“ vždy vyžaduje existenci/validaci plánu (tzn. v rámci UC „Spustit výrobní plán“ se vždy provede krok „vybrat/ověřit plán“).
Naplánovat výrobní zakázky <<include>> Ověřit kapacity výrobních zařízení (nový pod-UC)
Kapacitní kontrola je typicky povinná část plánování (stroj/kalendář/obsazenost).
Spustit výrobní zakázku <<include>> Ověřit připravenost výrobního zařízení (nový pod-UC)
Typicky check: zařízení dostupné, bez poruchy, správná konfigurace.
(Pozn.: tyto „ověřovací“ UC jsou velmi typické v zadáních, i když nejsou explicitně napsané — pomáhají ti udělat smysluplné include vazby.)
B) Monitoring a analýzy (tvůj příklad „monitoring → grafy“)
Tady je důležitá logika: grafy jsou forma zobrazení/analýzy, ale monitoring (sledování zakázky) je širší UC.
Doporučené vazby:

Monitorovat průběh výrobní zakázky <<include>> Zobrazit aktuální stav výrobní zakázky (nový pod-UC)
Stav = běží/pozastaveno/dokončeno, aktuální operace, kusy OK/NOK, čas.
Monitorovat průběh výrobní zakázky <<include>> Zobrazit grafy stavů výrobních zařízení
Pokud v rámci monitoringu zakázky systém vždy ukazuje i stav zařízení (OEE-like timeline) jako součást monitoringu.
Alternativa (často ještě čistší v UML):
Zobrazit grafy stavů výrobních zařízení je samostatný UC dostupný pro Zaměstnance
a Monitorovat průběh výrobní zakázky ho jen používá, pokud je to povinná součást. Když není povinná, pak to není include, ale spíš extend (ale ty teď chceš include, takže výše je „include-varianta“).
Analyzovat energetickou spotřebu výroby <<include>> Získat energetická data (nový pod-UC)
Získání dat (měření) je povinný krok analýzy.
Zobrazit grafy stavů výrobních zařízení <<include>> Získat data o stavech výrobních zařízení (nový pod-UC)
Opět: grafy bez dat nedávají smysl → datový sběr je povinný.
Pokud máš aktéra „Externí systém“, pak tyto „Získat data…“ UC mohou být napojené i na něj (Externí systém jako sekundární aktér dodávající data).

i cannot combine left to right in direction for parent and top to bottom direction just for the rectangle?

Short answer: No — PlantUML does not support mixed layout directions in nested scopes.
left to right direction and top to bottom direction are global graph layout hints for Graphviz, not scoped to containers like rectangle or package. When you place top to bottom direction inside rectangle MES, it does not create a local layout context; Graphviz still computes a single layout for the whole graph.
That’s why your code:

left to right direction
rectangle MES {
   top to bottom direction
}
does not actually isolate directions.
Practical solution (what you already almost did)
You must simulate local layout using constraints:
1. Use global direction
left to right direction
2. Force vertical stacking inside the system
Use hidden vertical links:
UC1 -[hidden]-> UC2
UC2 -[hidden]-> UC3
3. Force column ordering
Use hidden links between columns:
Employee -[hidden]-> MES
MES -[hidden]-> External
This is exactly the correct technique in PlantUML.
Cleaner trick (recommended for stable layouts)
Instead of linking random elements, use dummy anchors to stabilize columns.
Example:

@startuml
left to right direction

actor "Zaměstnanec" as Employee
actor "Externí systém" as External

rectangle MES {
  usecase UC1
  usecase UC2
}

' invisible anchors
() as LEFT
() as MID
() as RIGHT

LEFT -[hidden]-> MID
MID -[hidden]-> RIGHT

Employee -[hidden]-> LEFT
MES -[hidden]-> MID
External -[hidden]-> RIGHT
@enduml
This forces:
Actors  |  MES  |  System actors
and prevents Graphviz from reordering things.
Why PlantUML behaves like this
PlantUML delegates layout to Graphviz DOT, which:
Computes one directed graph
Has one global rank direction (rankdir)
Cannot nest rank directions.
So containers (package, rectangle) are visual only, not layout contexts.

Yes. In PlantUML you can influence how connection lines are routed using skinparam linetype. This directly affects how Graphviz draws the edges.
1. Orthogonal lines (usually best for UML diagrams)
skinparam linetype ortho
This forces 90° elbow connectors instead of curved/spline paths. It usually removes the “crazy spaghetti” routing.
Example:

@startuml
left to right direction
skinparam linetype ortho
@enduml
2. Polyline (less aggressive than default splines)
skinparam linetype polyline
This gives straight segments but not strict 90° angles.
3. Default (spline)
If you don't specify anything, Graphviz uses splines, which often produces messy curves when many edges exist.
Equivalent to:

skinparam linetype spline
Recommended settings for large UC diagrams
For your MES diagram, I would start with:
skinparam linetype ortho
skinparam shadowing false
skinparam packageStyle rectangle
This usually gives clean UML-style connectors.
Extra trick (very useful for UC diagrams)
You can force direction of arrows to avoid crossings:
-right->
-left->
-up->
-down->
Example:
UC5 -right-> UC13 : <<include>>
instead of:
UC5 ..> UC13
This dramatically stabilizes layout.
💡 Important limitation:
Even with these settings, PlantUML still relies on Graphviz automatic routing, so very dense diagrams can still look messy.

the line is going to some direction and than back to its target

Yes — that behavior is coming from Graphviz edge routing, not directly from PlantUML. When the layout engine tries to avoid overlaps it sometimes produces paths that go away from the target and then back, especially when:
many hidden edges constrain the layout
edges connect distant nodes across packages
the graph has conflicting direction hints
There are a few practical fixes.
1. Use orthogonal routing (most helpful)
Add this near the top:
skinparam linetype ortho
This forces 90° routing, which usually prevents the “loop away then back” effect.
Example:

left to right direction
skinparam linetype ortho
2. Explicitly control arrow direction
Instead of letting Graphviz decide:
UC5 ..> UC13 : <<include>>
force direction:
UC5 -right-> UC13 : <<include>>
or
UC5 -down-> UC13 : <<include>>
This often eliminates the weird detours.
3. Avoid mixing too many hidden constraints
You currently use many:
-[hidden]->
These create strong layout constraints. Too many can confuse Graphviz and cause strange routes.
Typical stable pattern:

hidden edges only for vertical stacking
dummy anchors only for columns
Avoid mixing both excessively.
4. If the problem is still present
Add:
skinparam nodesep 60
skinparam ranksep 60
This increases spacing between elements, giving Graphviz more room to route lines cleanly.
